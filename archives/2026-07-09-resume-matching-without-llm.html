Every AI project I have seen in the last two years starts the same way: someone opens a chat window, pastes in real data, gets an impressive answer, and concludes that the product should be built on a large language model. Ours started that way too. At the IT Consulting startup where I work, the recruiting team wanted a tool that could take a candidate’s resume and a job description and answer one question: how well does this person match this role?

The proof of concept was exactly what you would expect. We pasted sample resumes and sample job descriptions into cloud LLM APIs and asked for a match assessment. The answers read beautifully. And then three problems surfaced, none of which had anything to do with answer quality:

  1. Cost didn’t align with how we actually pay for things. As a startup, we run on Azure credits. Anything that executes inside our tenant — compute, storage, databases — is effectively covered by spend we have already committed. Third-party LLM APIs are a separate invoice, in real cash, that grows per comparison. Ranking every candidate against every open requisition is a multiplicative workload: 200 resumes against 10 job descriptions is 2,000 LLM calls, and that’s one re-rank. The bill wasn’t hypothetically large; it was structurally misaligned with where our money already was.

  2. Candidate PII was leaving our boundary. Resumes are dense with personal data — names, addresses, phone numbers, employment history. Sending them to a third-party model provider puts a vendor in the data path, with everything that implies: data processing agreements, retention questions, and a compliance story that gets harder in regulated industries. We wanted candidate data to never leave our own cloud tenant.

  3. The scores weren’t defensible. Ask an LLM to score the same resume against the same job description twice and you can get two different numbers, each wrapped in fluent, confident prose. For a hiring tool, that is not a cosmetic flaw. If a recruiter — or a regulator — asks why did this candidate score 72?, “the model felt that way at that temperature” is not an answer.

We also looked at commercial resume-matching products, and found what most small companies find: they are priced and packaged for enterprise HR departments, not for a startup that wants one focused workflow.

So we stopped and asked a more basic question: what is this task, actually? And the answer changed the architecture entirely. Matching a resume to a job description is a ranking problem, not a generation problem. Nothing needs to be written. No prose needs to be produced. We need a similarity signal, a skill overlap signal, and a deterministic way to combine them into a score a human can interrogate.

That realization let us delete the LLM from the design. What we shipped instead is a two-container system built on a 110M-parameter sentence embedding model, cosine similarity, and Azure SQL — no generative model, no GPU, no dedicated vector database, and no byte of candidate data leaving our tenant. This post walks through how it works.

Who this is for: engineers who have been told to “add AI” to an internal tool and suspect the task might not need a generative model; teams in regulated or privacy-sensitive domains who want AI-assisted workflows without a third-party model vendor in the data path; and anyone curious what Azure SQL’s native VECTOR type looks like in production-shaped code. All code in this post has been rewritten and simplified for illustration — it shows the real algorithms, not the verbatim production source.

The Obvious Objection: Why Not Azure OpenAI?

Before the architecture, the question any Azure shop will ask first. If the constraints are “burn Azure credits” and “keep data in the tenant,” Azure OpenAI Service satisfies both: it bills against Azure spend, runs inside your subscription boundary, and Microsoft contractually does not train on your data.

We could have used it, and for a generation-shaped task we would have. But it does nothing for the third problem. A generative model — wherever it runs — gives you non-deterministic scores and explanations that are narratives about the score rather than derivations of it. Hiring is a domain where “show your work” is not optional (more on the regulatory angle later). Once we accepted that the task was ranking, not generation, the deciding pillar was determinism and explainability, and no deployment model of a generative LLM provides that. An embedding model does, trivially: the same text produces the same vector, every time, and every point of the final score traces to either a named cosine similarity or a named missing skill.

Architecture Overview

The whole system is two containers and a database:

    API --> DB
  • FastAPI backend owns every piece of logic: parsing DOCX/PDF documents into sections, embedding them, extracting skills, computing scores, and rendering PDF scorecards. The embedding model — sentence-transformers/all-mpnet-base-v2, 768 dimensions, CPU-only — is downloaded at image build time and baked into the container. At runtime, nothing calls out to anything. The container runs comfortably on 2 vCPU / 4 GB on Azure Container Apps.
  • Streamlit UI is a deliberately thin client. Zero business logic; every action is a REST call to the API. Recruiters authenticate via Azure AD OAuth2. Keeping the UI dumb means the API is the single place where behavior lives — and the UI layer is swappable when Streamlit stops being enough.
  • Azure SQL is both the application database and the vector store. Section embeddings live in native VECTOR(768) columns next to the relational data they describe. No Pinecone, no Qdrant, no second system to operate.

There is a fourth participant: the company’s existing web portal pushes candidate uploads directly into the API through a dual-authenticated endpoint (API key plus Azure AD bearer token), so resumes flow in without anyone re-uploading them.

The data path is the point of this diagram: every arrow starts and ends inside the tenant. There is no arrow leaving the box.

Deep Dive 1: Section-Aware Embedding and Best-Match Cosine

The naive approach — embed the whole resume, embed the whole job description, take one cosine similarity — produces one mushy number. Resumes and JDs are both structured documents wearing unstructured clothes: a JD has responsibilities, required qualifications, preferred qualifications; a resume has experience, skills, education, projects. A single whole-document vector averages all of that into semantic soup, and a candidate with a stellar skills section and an irrelevant hobbies section looks the same as a uniformly mediocre one.

So we never embed whole documents. At upload time, documents are split into sections (heading-style detection for DOCX, capitalization heuristics for PDF — more on the brittleness of that later), and each section is embedded and stored individually:

from sentence_transformers import SentenceTransformer

model = SentenceTransformer("sentence-transformers/all-mpnet-base-v2")

def embed(text: str):
    # normalize_embeddings=True → unit vectors, so cosine similarity
    # later reduces to a plain dot product
    return model.encode(text, normalize_embeddings=True)

That normalize_embeddings=True is doing quiet work: with unit-length vectors, cosine similarity is just np.dot(a, b) — no norms to compute at query time.

The scoring side is where the design earns its keep. The obvious follow-up mistake would be to match sections by name — compare the JD’s “Requirements” to the resume’s “Requirements.” But resumes don’t have a Requirements section, and no two resumes name their sections the same way. Instead, for each JD section, we take the best cosine similarity across all candidate sections, wherever it is found:

# Simplified for illustration
def semantic_score(jd_sections, cand_sections, weights):
    """jd_sections / cand_sections: list of (name, unit_vector)."""
    weighted_sum, total_weight = 0.0, 0.0

    for jd_name, jd_vec in jd_sections:
        # Best match across ALL candidate sections — section names
        # never need to line up
        best = max(np.dot(jd_vec, cand_vec) for _, cand_vec in cand_sections)

        weight = resolve_weight(jd_name, weights)   # e.g. "requirements" → 0.35
        weighted_sum += best * weight
        total_weight += weight

    return weighted_sum / total_weight

This is section-name-agnostic by construction. If the JD’s “Required Qualifications” content is best answered by the resume’s “Professional Experience” section, that’s the pairing that counts — and we record which candidate section won, because that pairing is part of the explanation shown to the recruiter.

The weights come from a section_weights table that maps keywords to importance (“requirements” sections matter more than “about the company” boilerplate), and the recruiting team can tune them from an admin screen. When weights change after a ranking was computed, the UI shows a staleness banner prompting a re-run — a small feature that prevents a large class of “why do these numbers disagree” confusion.

The asymmetry in the loop is deliberate: we iterate over JD sections, not resume sections. The job description defines what matters; the resume is searched for evidence. A resume section that matches nothing in the JD simply contributes nothing — it isn’t penalized, it’s ignored.

Deep Dive 2: An Explainable Composite Score

Semantic similarity alone over-rewards fluent, keyword-adjacent writing. A resume can sound like a job description without the candidate ever having used the required tools. So the final score is a composite of two components:

overall = 0.60 × semantic + 0.40 × skill

The skill component is deterministic set arithmetic over skills extracted at upload time (extraction itself is covered below). Three rules give it its shape:

  1. Required skills are worth 3× preferred skills. A JD skill found under a “Required” / “Must have” / “Essential” header carries triple weight. Missing a preferred skill dents the score; missing a required one craters it.
  2. Experience gaps are penalized, not zeroed. If the JD asks for 5 years of Python and the candidate has 3, the match still counts — reduced by 15% per missing year, floored at 50%. A floor matters: a strong candidate one year short of an arbitrary threshold should be visible, not filtered out.
  3. No skill data, no penalty. If either side lacks a years figure, the multiplier is 1.0. The system never punishes a candidate for information the parser failed to find.

# Simplified for illustration
REQUIRED_MULTIPLIER = 3
PENALTY_PER_YEAR = 0.15
PENALTY_FLOOR = 0.5

def years_penalty(jd_years, cand_years):
    if jd_years is None or cand_years is None:
        return 1.0                        # missing data → benefit of the doubt
    gap = max(0, jd_years - cand_years)
    return max(PENALTY_FLOOR, 1.0 - gap * PENALTY_PER_YEAR)

def skill_score(jd_skills, cand_skills):
    """jd_skills: {skill_id: {required: bool, years: int|None}}
       cand_skills: {skill_id: {years: int|None}}"""
    total = sum(REQUIRED_MULTIPLIER if s["required"] else 1
                for s in jd_skills.values())
    if total == 0:
        return 1.0

    earned = 0.0
    for skill_id, jd_info in jd_skills.items():
        if skill_id in cand_skills:
            worth = REQUIRED_MULTIPLIER if jd_info["required"] else 1
            earned += worth * years_penalty(jd_info["years"],
                                            cand_skills[skill_id]["years"])
    return earned / total

Every constant in that snippet — the 60/40 split, the 3× multiplier, the 15% penalty, the 50% floor — is a policy decision sitting in plain sight, arguable and changeable. That is the property we could never get from a generative model. When a recruiter opens a candidate’s scorecard, they don’t see a paragraph of AI prose; they see: this JD section matched that resume section at 0.81, these four required skills matched, this one is missing, this one matched with a two-year experience gap. Score the same pair tomorrow, get the same number. Change a weight, and the change is visible, versioned, and explainable.

The output is a structured result — overall score, both components, per-section pairings with similarities, and a full skill breakdown (matched / missing-required / missing-preferred / extra) — which drives both the UI detail view and a generated PDF scorecard.

Deep Dive 3: Azure SQL as the Vector Store

The reflexive move in 2026 is to add a vector database the moment embeddings appear in a design. We didn’t, for a boring reason: Azure SQL (and SQL Server 2025) has a native VECTOR type, and our scale — thousands of candidates, not millions — doesn’t need approximate nearest-neighbor indexes. Every vector we need for one ranking fits in memory of a small container.

# INSERT: cast the JSON parameter to VECTOR(768)

    jd_id, name, order, text, vector_to_sql(vec),

# SELECT: cast back to NVARCHAR and parse

    "SELECT section_name, CAST(embedding AS NVARCHAR(MAX)) "

sections = [(row[0], np.array(json.loads(row[1]), dtype=np.float32))

Is JSON-serializing floats elegant? No. Does it matter at this scale? Also no — a full ranking of every active candidate against a JD is a few hundred small reads followed by NumPy dot products, and it completes interactively.

What we bought by not adding a vector database: one system to secure, back up, and grant access to; vectors that live transactionally next to the rows they describe (delete a candidate, their embeddings go with them — GDPR erasure is one DELETE); and the entire compliance story of “where is candidate data?” answered with “the same Azure SQL database as before.” Azure SQL also has a native VECTOR_DISTANCE() function for pushing similarity into the database; we compute in Python instead because the best-match-per-section logic and the audit trail of section pairings are easier to express there, and the data volumes make the transfer cost irrelevant. At a million candidates, this section of the post would be written very differently — that’s the honest boundary of the design.

The Rest of the Pipeline, Briefly

Three more components deserve a paragraph each.

Skill extraction runs once at upload time, never at query time, against a curated skill taxonomy with aliases (React = ReactJS = React.js). Three passes, cheapest first: exact whole-word regex; then a normalized pass that strips punctuation and spacing (so Node.js, NodeJS, and “Node JS” collapse); then a fuzzy pass via rapidfuzz at an 88% similarity threshold for the typos. JD skills found under “Required” / “Must have” / “Essential” headers get the required flag, and a regex scan of the ±100 characters around each mention pulls out “N+ years” patterns. Extracted skills are stored relationally, which is what makes the skill score pure set arithmetic later.

Section detection is honest heuristics, not magic. DOCX files offer real structure — heading styles first, bold-line detection as fallback. PDFs offer almost none, so we detect section boundaries by capitalization and title-case patterns. This works on conventionally formatted documents and degrades on creative layouts; it is the most brittle part of the system and the first thing we’d upgrade (a small local layout model could slot in here without changing anything downstream).

The workflow layer is what turned a scoring engine into a tool the team actually uses: per-JD candidate status (pending → shortlisted / rejected / interviewed / offered) with notes and audit fields, status badges in the ranked list, and the staleness banner mentioned earlier when taxonomy or weights change after a ranking. None of it is technically interesting. All of it is why the tool gets opened.

The Part Most AI-Recruiting Posts Skip: This Is a Regulated Category

Automated tools that influence hiring decisions are not a neutral engineering playground. New York City’s Local Law 144 requires bias audits for automated employment decision tools. The EU AI Act classifies AI used in recruitment as high-risk. The EEOC has published guidance on algorithmic employment assessments. If you are building in this space, this is design input, not legal trivia.

This context reframes the determinism pillar from an engineering preference into a posture:

  • Reproducible: the same resume and JD produce the same score, every run. There is no temperature, no sampling, no “it said something different yesterday.”
  • Decomposable: every score factors into named cosine similarities and named skill matches with visible weights. If a scoring policy is challenged, the policy is right there in the constants — and changing it is a diff, not a re-prompt.
  • Human-in-the-loop by construction: the system ranks; it never rejects. Every status change — shortlist, reject, interview — is a recruiter’s explicit action, recorded with who and when. The tool orders a reading list; humans make decisions.

I want to be precise about the claim: this system is designed to be auditable — it has not undergone a formal bias audit. Embedding models carry the biases of their training data, and a deterministic pipeline can encode a biased policy deterministically. What the architecture guarantees is that when an audit happens, there is something concrete to audit: fixed weights, traceable scores, and a recorded human decision at every consequential step. A generative scorer can’t offer that even in principle.

What This Doesn’t Do

An architecture post that hides its limitations is an ad. Ours, honestly:

  • Validation is spot-check only so far. We’ve eyeballed scores across sample resumes and they rank sensibly, but we have not yet run a structured comparison against recruiter judgment (e.g., “does the top 5 match who a human would shortlist?”). That is the next milestone, and until it’s done, claims about matching quality stay modest.
  • Embeddings don’t understand negation or recency. “Migrated away from Java” and “expert in Java” embed uncomfortably close together, and “used Python in 2015” reads like “uses Python daily.” The years-extraction mitigates some of this for skills; the semantic component remains exposed to it.
  • Section detection is heuristic. Creative resume layouts, tables, and multi-column PDFs degrade chunking quality, and bad chunks mean bad section scores.
  • The taxonomy is a garden. Skill extraction is only as good as the skill list and its aliases, and that requires ongoing human curation. The admin CRUD exists precisely because this never becomes “done.”
  • There’s no reranking stage. A cross-encoder pass over the top-N candidates would likely sharpen the head of the ranking at trivial cost. It’s the most obvious upgrade that preserves every property we care about — still deterministic, still local, still explainable.

Could This Run Fully On-Prem?

A closing note for readers in finance, insurance, and healthcare who read “stays in our tenant” and thought “not good enough.” The primary claim of this build is the tenant boundary: candidate PII never touches a third-party AI vendor. But notice what the architecture doesn’t depend on: no GPU, no external API, no managed AI service. The embedding model is 420 MB of files loaded by a CPU process. Everything except the database would move to an on-prem Docker host unchanged.

The one honest caveat is Azure SQL’s VECTOR type, which doesn’t exist in on-prem SQL Server 2022 (it arrives with SQL Server 2025). A true air-gapped port today would swap in PostgreSQL with pgvector — a change confined to the serialization helpers and a handful of SQL statements, because the similarity math already lives in Python. The design is cloud-resident by choice, not by dependency.

Takeaways

  1. Name the task before choosing the model. “AI that matches resumes to jobs” sounds like an LLM problem. It’s a ranking problem, and ranking needs similarity and scoring, not generation. Asking “what does this task actually produce?” deleted the most expensive and least explainable component from our design.
  2. Align architecture with how you already spend. For a startup on cloud credits, in-tenant compute is effectively pre-paid and third-party API invoices are new cash. That framing — not abstract per-token math — is what made the cost decision obvious.
  3. In regulated workflows, determinism is a feature you can’t retrofit. Reproducible, decomposable scores with a human on every decision is an architecture property. No amount of prompt engineering adds it to a generative scorer afterward.
  4. Your database might already be your vector store. At thousands-of-documents scale, native vector columns next to your relational data beat operating a second system — and inherit its access control, backups, and deletion semantics for free.
  5. A 110M-parameter encoder on a CPU is more model than most internal tools need. The gap between “impressive demo” and “deployable system” was not closed by a bigger model. It was closed by a smaller one, plus arithmetic we can defend.
