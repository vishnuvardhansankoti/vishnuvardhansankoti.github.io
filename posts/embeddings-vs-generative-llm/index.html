Embeddings vs. a Generative LLM for Document Matching: The Engineering Case

Resume-to-job-description matching looks like an LLM task: two documents go in, a judgment comes out. We built it without one. The system that shipped uses a 110M-parameter sentence embedding model (all-mpnet-base-v2), cosine similarity, and a deterministic scoring function — no generative model anywhere in the pipeline.

This post is the engineering argument for that choice. No war stories, no compliance discussion — just what the embeddings-only architecture buys you over a generative LLM when the task is scoring and ranking documents against each other, and where the trade goes the other way.

The Two Architectures

Generative approach: for each resume/JD pair, send both documents to an LLM with a prompt like “score this candidate against this job description and explain your reasoning.” The model reads both documents and produces a score and a narrative, per pair, per run.

Embeddings approach: at upload time, split each document into sections and embed each section once. At scoring time, compute cosine similarities between stored vectors, combine with an exact skill-overlap score, and emit a composite:

# Per JD section, take the best cosine across all candidate sections,
# then weight by section importance. Vectors are unit-normalized,
# so cosine similarity is a plain dot product.
def semantic_score(jd_sections, cand_sections, weights):
    weighted_sum, total_weight = 0.0, 0.0
    for jd_name, jd_vec in jd_sections:
        best = max(np.dot(jd_vec, cand_vec) for _, cand_vec in cand_sections)
        w = resolve_weight(jd_name, weights)
        weighted_sum += best * w
        total_weight += w
    return weighted_sum / total_weight

overall = 0.6 * semantic_score(...) + 0.4 * skill_score(...)

Everything below follows from the structural difference between these two designs.

Advantage 1: Determinism

An embedding model is a pure function. The same text produces the same vector on every run, so the same resume/JD pair produces the same score today, tomorrow, and after a container restart. There is no temperature, no sampling, no run-to-run variance to explain away.

A generative scorer is a distribution, not a function. Even at temperature 0, provider-side model updates and floating-point nondeterminism can shift outputs between runs — and any score that can silently change between identical invocations is a score you cannot regression-test. With embeddings, score(A, B) is as testable as any other function in the codebase: fixtures in, exact expected values out, breaks loudly in CI when someone changes a weight.

Advantage 2: Explainability That Is a Derivation, Not a Narrative

Ask a generative LLM why it scored a candidate 72 and it will produce a fluent paragraph. That paragraph is a plausible story generated after the score — there is no mechanical guarantee it reflects whatever produced the number.

The embeddings pipeline’s explanation is the computation:

  • This JD section matched that resume section at cosine 0.81, with weight 0.35.
  • These four required skills matched; this one is missing; this one matched with a two-year experience gap, applying a 0.70 multiplier.
  • 0.6 × semantic + 0.4 × skill = the score, reproducible to the decimal.

Every point in the final number traces to a named similarity or a named skill, and every policy constant (the 60/40 split, the 3× required-skill multiplier, the per-year penalty) sits in code where it can be read, argued about, and changed with a diff. When the scoring policy needs to change, it is a code review — not a prompt-engineering session followed by hopeful re-evaluation.

Advantage 3: The Cost Curve Bends the Other Way

Ranking is a multiplicative workload. N resumes against M job descriptions is N×M comparisons, and every taxonomy tweak or weight change triggers a re-rank.

  • Generative: every comparison re-reads both full documents through the model. Cost is per-token, per-pair, per-run — 200 resumes × 10 JDs = 2,000 LLM calls, each processing a few thousand tokens, every time you re-rank.
  • Embeddings: each document is embedded exactly once, at upload. A comparison after that is a handful of 768-dimensional dot products — microseconds of CPU. Re-ranking the entire corpus costs effectively nothing, forever.

The generative design couples your marginal cost to usage; the embeddings design front-loads a one-time cost per document and makes queries free. For any workload where documents are compared many times (which is what “ranking” means), the curves diverge fast.

Advantage 4: Latency and Throughput

A generative comparison takes seconds — the model must ingest both documents and autoregressively produce its judgment, one token at a time. Ranking a few hundred candidates serially is minutes; doing it in parallel is a rate-limit negotiation.

The embeddings comparison is a memory read plus NumPy dot products. Ranking every active candidate against a JD completes interactively, on 2 vCPU, with no rate limits because there is nothing to rate-limit. Interactive re-ranking changes how the tool gets used: adjusting a section weight and immediately seeing the ranking shift is a feedback loop a batch-of-LLM-calls design cannot offer.

Advantage 5: Nothing Leaves the Box

The embedding model is ~420 MB of files loaded by a CPU process, baked into the container image at build time. At runtime the system makes zero external calls: no third-party API in the data path, no vendor seeing document contents, no retention policy to review, no dependency on someone else’s uptime or deprecation schedule. The privacy boundary is the deployment boundary — and the same property makes the stack portable to any Docker host, air-gapped included.

A generative design — API-based or even self-hosted behind an API layer — puts a network hop and usually a vendor between your documents and your score.

The last row deserves a note. The embeddings pipeline fails legibly: if section detection mangles a PDF or a skill alias is missing from the taxonomy, the gap is inspectable in the stored data. A generative scorer fails fluently — inventing a skill match or drifting from the requested output format while sounding entirely confident — and fluent failures are the expensive kind, because nothing looks wrong.

Where a Generative LLM Would Win

The honest half of the comparison — the trade has real costs:

  • Nuance and negation. Embeddings score “migrated away from Java” uncomfortably close to “expert in Java,” and weigh “used Python in 2015” like “uses Python daily.” A capable LLM reads both correctly. We mitigate with explicit years extraction, but the semantic component stays exposed.
  • Zero-shot flexibility. An LLM handles a new document type, an unusual JD format, or a “compare these on communication skills” request with a prompt change. The embeddings pipeline needs code: chunking rules, taxonomy entries, weight configuration.
  • No curation burden. The skill taxonomy and its aliases require ongoing human gardening. An LLM ships with world knowledge; our pipeline ships with a CRUD admin page.
  • Narrative output. If the product requirement is a written assessment — an actual paragraph a recruiter sends to a hiring manager — you need generation, by definition.

The pattern in that list: the LLM wins where the task genuinely requires reading comprehension or writing. It does not win at producing a stable, auditable number — which is what ranking asks for.

There is also a middle path worth naming: keep the embeddings pipeline as the ranking engine and add a cross-encoder reranker over the top-N — still deterministic, still local, sharper at the head of the ranking. And if narrative summaries are ever needed, generate them from the structured score breakdown rather than letting a generative model produce the score itself. Generation as a presentation layer over deterministic scoring keeps every advantage above.

The Takeaway

Choose by output type, not by hype cycle. If the task’s output is prose, you need a generative model. If the output is a number used to order things — matching, ranking, deduplication, recommendation, triage — then a small embedding model plus explicit scoring code gives you determinism, derivable explanations, near-zero marginal cost, interactive latency, an airtight data boundary, and a system you can unit-test. For document matching, the 110M-parameter encoder wasn’t the compromise. It was the correct tool.
