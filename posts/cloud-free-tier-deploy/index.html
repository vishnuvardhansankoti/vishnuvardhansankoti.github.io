Deploy a Full App Stack for Nearly Free: What the Major Cloud Providers Actually Give You

With AI-assisted coding making it easier than ever to build working apps, the bottleneck has shifted: not building the app, but putting it somewhere other people can actually use it. A solo developer or a small team can now go from idea to working prototype in a weekend. The problem starts on Monday, when they open a cloud provider’s pricing page.

Those pages are not designed to help you spend less. They are organized around products, not use cases — dozens of service tiers, interlinked dependencies, pricing calculators that require you to already know what you want. The natural response is to close the tab and either leave the project on localhost or pay for something familiar like a $20/month Heroku equivalent, on the assumption that production-grade deployment just costs money.

That assumption is mostly wrong, at least for the first phase of any project.

Azure, AWS, and GCP each maintain a set of always-free services — not trials, not credits, not “free for 90 days” promotional offers, but permanent free limits that reset monthly and never expire. They are buried under the marketing, but they exist and they are genuinely generous. A full-stack application — React frontend, REST API backend, relational database, and a CI/CD pipeline — can run inside the free tier of one or more providers at low-to-moderate traffic with a monthly cloud bill of zero.

The difficulty is knowing where exactly those limits sit for each layer of your stack. The frontend story is different from the database story. The VM story is different from the serverless story. And the “always free” story is very different from the “12-month trial that becomes a bill you forgot about” story. Cloud providers do not make these distinctions obvious, which is how developers end up surprised by charges on services they thought were free — or avoid the cloud entirely because the pricing looks too complex to navigate.

This article maps the free tier limits for every layer of a typical full-stack app — frontend, backend, virtual machines, containers, CI/CD, relational database, NoSQL database, and ML/data analytics — across all three major cloud providers. Each section explains what you actually get, where the limits bite, and what to do when they do. Where the free tier genuinely runs out (primarily relational databases and container registries), the cheapest paid option is included so you know the real cost floor before committing to a stack.

Who this is for: developers who have built something and want to ship it; junior-to-mid engineers setting up their first production deployment; and senior engineers who know how to deploy but have never audited what they could be running for free. The numbers in this article are current as of mid-2026 — cloud providers do adjust free tier limits, so the reference links in each table are the authoritative source.

A note on free tier types before the numbers: cloud providers use two distinct categories. Always Free means the limit resets monthly and never expires. 12-Month Free means the limit applies to new accounts for the first year only — after that, you pay. The distinction matters. Decisions built on a 12-month free tier that aren’t revisited before expiration are how unexpected bills appear.

1. Frontend — Static Hosting

The Problem

A React, Vue, or plain HTML/CSS application compiled to a dist/ or build/ folder is a collection of static files. Serving static files is one of the cheapest things a cloud provider can do, and it shows in how generous their free tiers are. The mistake most developers make is defaulting to a full application server — an EC2 instance, a VM, a container — to serve what is essentially a folder of files. You pay for idle compute when a CDN would serve the same files faster and for free.

The Solution

All three major providers offer static hosting with CDN delivery. GitHub Pages is included as a fourth option — it is the simplest path for public projects and has no cloud account requirement.

Azure Static Web Apps is the most complete free option for a production app. It includes a built-in serverless API layer (backed by Azure Functions), GitHub Actions CI/CD integration, custom domain support with auto-provisioned SSL, and staging environments — all on the free plan. The 100 GB/month bandwidth limit handles thousands of daily visitors without issue.

AWS CloudFront offers 1 TB/month of free data transfer out, permanently — the most generous bandwidth limit of the group. The catch is that S3 (the underlying storage) is only free for 12 months on new accounts. After that, S3 storage for static sites is inexpensive ($0.023/GB/month), but it is no longer free.

Firebase Hosting has the largest included storage (10 GB) but its bandwidth allowance — 360 MB per day — is a hard ceiling that a modest traffic spike can hit in hours. It suits apps with steady, low traffic, not anything likely to get a sudden surge.

GitHub Pages is the easiest starting point. Push a branch, enable Pages in repository settings, and the site is live. The 1 GB storage limit is tight for image-heavy sites, but 100 GB/month of bandwidth means traffic is not the constraint.

Limitations

Azure Static Web Apps’ free plan limits you to 10 apps per subscription and restricts API execution time to 45 seconds per invocation. For most apps, neither limit is relevant — but if you are building multiple projects under one Azure account, the 10-app ceiling matters.

GitHub Pages is available for public repositories only on free GitHub accounts. It serves static files exclusively — no server-side logic, no API routes, no environment variables at runtime.

Firebase’s daily bandwidth cap (360 MB) rather than monthly is unusual. An app that gets mentioned on a popular forum can breach the daily limit within minutes. Budget accordingly or upgrade to the Blaze (pay-as-you-go) plan if the site is externally linked.

2. Backend — Serverless Functions

The Problem

A REST API or batch job running in a full container or VM wastes most of what it pays for. An API handling 10 requests per second runs for a few milliseconds of compute per request — the other 999 milliseconds of each second, the server sits idle, billing by the hour. Serverless execution inverts this: you pay only for the milliseconds the function actually runs. At low to moderate request volumes, the free tier covers everything.

The Solution

Serverless functions are where the always-free tiers are most generous. All three providers offer permanently free compute limits large enough to run a production API at modest scale.

AWS Lambda + API Gateway is the most widely documented serverless stack. Lambda handles function execution; API Gateway exposes it as an HTTP endpoint. Both have permanent 1 million free invocations per month. A typical REST API with 10 endpoints handling 100 requests/day runs under 300,000 invocations per month — comfortably inside the free tier. Lambda also accepts a free Function URL (no API Gateway needed) for simpler use cases.

Azure Functions matches Lambda on compute (400,000 GB-seconds) and free invocations (1 million). If your frontend is already on Azure Static Web Apps, Functions is the natural backend companion — the Static Web Apps free plan includes a built-in Functions integration that requires no separate configuration.

GCP Cloud Functions (1st gen) offers twice the free invocations (2 million/month) for the same compute budget. One important caveat: Cloud Functions 2nd gen (based on Cloud Run) does not share the same always-free tier as 1st gen. If you use 2nd gen, you are in pay-as-you-go territory from the first invocation.

Limitations

The GB-seconds metric is a product of memory allocation and execution duration. A function configured with 512 MB of memory that runs for 1 second consumes 0.5 GB-seconds. At 400,000 GB-seconds free per month, that function can run 800,000 times per month — well beyond the 1 million invocation cap. In practice, invocations are usually the binding constraint, not compute time.

Cold start latency — the delay when a function initializes after a period of inactivity — affects all three platforms. For interactive APIs, this produces occasional response times of 500ms to 2 seconds on the first request after a quiet period. This is a trade-off of the serverless model, not a platform bug. If consistent sub-100ms response times are required, keep a warmed container (see the next section) or use provisioned concurrency (which is paid).

3. Virtual Machines

The Problem

Containers and serverless functions handle the majority of backend workloads well — but there are cases where you need the full surface of an operating system. Self-hosted databases, game servers, VPN endpoints, continuous background daemons, legacy applications that cannot be containerized, and anything that requires kernel-level configuration all need a VM. The challenge is that a VM is always-on by definition: unlike a serverless function or a scaled-to-zero container, it runs continuously whether serving traffic or sitting idle, and the billing reflects that.

The free tier picture for VMs is thinner than any other category among Azure, AWS, and GCP — all three offer VM free tiers limited to 12 months. Oracle Cloud Infrastructure (OCI) is the exception that makes this section worth covering: it provides the only genuinely always-free VM allocation among major cloud providers, and its ARM Ampere instances are meaningfully more capable than a t2.micro.

OCI Ampere A1 is the standout option in this category. 2 OCPUs and 12 GB of RAM — configurable across up to 4 instances — is enough to run a small Kubernetes cluster, a self-hosted database, or a backend service with a sidecar. Arm-based workloads are broadly supported by modern Linux distributions and all mainstream server runtimes (Node.js, Python, Go, Java, .NET). The free tier does not expire.

In addition to the A1 ARM instances, OCI also provides two Always Free AMD Micro VMs (VM.Standard.E2.1.Micro), each with 1/8 OCPU and 1 GB RAM. These are suitable for lightweight daemons or utility processes, not for serving application traffic.

GCP e2-micro is the only always-free x86 VM among the big three cloud providers. One instance per billing account, restricted to US regions (us-central1, us-east1, us-west1). For US-based applications this is rarely a practical limit. For anything requiring a non-US region, the e2-micro free tier is unavailable.

AWS t3.micro and Azure B1s are structurally identical in what they offer: 750 hours per month of a single small VM for the first 12 months after account creation. At exactly one instance running 24/7, the 750 hours covers a full month. After 12 months, both cost approximately $7–9/month. These are useful for getting started but are not a long-term free option.

Limitations

OCI’s Ampere A1 free tier was reduced in June 2026 from 4 OCPUs/24 GB to 2 OCPUs/12 GB for always-free accounts — verify your current allocation in the OCI console if you sized workloads against the old limit. Pay-as-you-go OCI accounts may still receive the higher allocation; the reduction targets accounts on the always-free plan specifically.

OCI’s free tier is also subject to availability constraints. Provisioning an A1 instance in high-demand regions — us-ashburn-1 is the most affected — frequently returns “Out of Capacity” errors even for free-tier accounts. Retry at off-peak hours or switch to a less-contested region (eu-frankfurt-1, ap-sydney-1) if provisioning fails repeatedly.

VMs do not scale to zero. An idle OCI A1 or GCP e2-micro VM continues to consume its free allocation around the clock. There is no monetary cost, but the capacity is occupied — you cannot bank unused compute or provision a different VM shape while the allocation is held. Delete free VMs you are not actively using rather than leaving them idle; this frees the allocation for a different configuration when you actually need it.

4. Containers — Backend API and Registry

The Problem

Not every backend fits neatly into a serverless function. Long-running processes, batch jobs, WebSocket servers, ML inference endpoints, and applications with complex startup dependencies are difficult to shoehorn into a stateless function model. Containers are the natural fit — but running a container on a VM means paying for a VM around the clock, even when idle. Container-as-a-service platforms solve this by scaling to zero: the container shuts down when there is no traffic and starts again on the next request.

Before any container can run in the cloud, it needs to be stored somewhere. Container registries are the equivalent of npm publish for Docker images — push an image, reference it by tag, pull it at deployment time.

GitHub Container Registry (GHCR) is the practical default for most projects: unlimited storage for public images, free pulls, and no cloud account required beyond a GitHub account. If your repository is already on GitHub, storing your image in GHCR adds no overhead and eliminates the registry cost entirely. Private GHCR images count against your GitHub Actions storage quota (500 MB on free accounts).

AWS ECR Public offers 50 GB/month of free storage for public repositories — enough for dozens of application images at typical sizes. Private ECR storage is free for only 12 months on new accounts.

Azure Container Registry has no free tier. The cheapest option is the Basic SKU at approximately $5/month, which includes 10 GB of storage. This is the one unavoidable line item that makes an otherwise free Azure deployment cost something from day one.

Container Hosting

Scale-to-zero is supported by both GCP Cloud Run and Azure Container Apps, meaning the container shuts down when idle and cold-starts on the next request. AWS has no equivalent always-free hosted container service.

GCP Cloud Run is the strongest free offering for containerized backends. 2 million free requests per month with scale-to-zero means a container-based API runs entirely free at low traffic volumes — and you pay only when the container is actually handling requests. The free tier is per billing account and resets monthly.

Azure Container Apps matches Cloud Run’s free compute limits exactly (180,000 vCPU-seconds, 360,000 GB-seconds per month) and also scales to zero. The free tier covers a backend API handling several thousand requests per day without cost.

AWS has no always-free tier for containerized workloads. App Runner and ECS Fargate are pay-as-you-go from the first second. For AWS users who need containers, Lambda with container image support (Lambda accepts container images up to 10 GB) is the viable alternative that stays inside Lambda’s always-free tier.

Limitations

Scale-to-zero has the same cold start trade-off as serverless functions. When a Cloud Run or Container Apps instance has scaled to zero and a new request arrives, the container must start before the request is served. For containers with heavier runtimes — Java, .NET with reflection-heavy startup, Python with large import trees — this cold start can exceed 5–10 seconds. Go and Node.js start faster and are better suited to scale-to-zero deployments.

The free compute limits on Cloud Run and Container Apps are based on CPU and memory allocation, not wall-clock time. A container allocated 1 vCPU and 512 MB that runs for 100 seconds consumes 100 vCPU-seconds and 50 GB-seconds. At 180,000 vCPU-seconds free per month, that container can handle roughly 1,800 such 100-second runs — at realistic API request sizes (each completing in under a second), the effective free tier covers well over 100,000 requests per month.

5. CI/CD — Deployment Pipelines

The Problem

A deployment pipeline is the bridge between code and a running application. Without one, every deployment is a manual sequence of steps: build the project, run tests, push a container image, update the hosting environment. Done by hand, this sequence is slow, error-prone, and skipped under deadline pressure — which is exactly when it matters most. CI/CD automates the full path from a git push to a live deployment, and the free tier limits here are among the most practically useful in this entire article, because getting CI/CD wrong costs you time on every single change.

The options split into two categories: hosted runners (the CI/CD provider runs the build machine, you pay by the minute) and self-hosted runners (you provision a cloud VM — an OCI A1 instance, a GCP e2-micro, or any other always-free VM from the previous section — install the runner agent, and minutes are unlimited). The runner agent runs as a background service on the VM and polls GitHub or Azure DevOps for queued jobs over outbound HTTPS — no inbound ports, no local machine required. For low-traffic side projects, hosted runners inside the free tier cover everything. For teams or high-volume pipelines, a self-hosted runner on a free cloud VM eliminates the minute constraint entirely.

GitHub Actions with public repositories is unlimited at zero cost — the most generous CI/CD offer in the industry. Every push, pull request, and scheduled run triggers a fresh Ubuntu, Windows, or macOS runner with no minute counter. For open-source projects or anyone willing to work in public repositories, the free tier is effectively infinite.

GitHub Actions on private repositories allocates 2,000 Linux minutes per month on the free plan, with 500 MB of artifact storage. 2,000 minutes covers roughly 66 daily pipeline runs averaging 30 minutes each — sufficient for a solo developer or a small team. macOS runners consume minutes 10× faster than Linux, and Windows runners 2× faster; use Ubuntu runners unless the build specifically requires another OS.

Azure DevOps Services grants 1,800 minutes per month (30 hours) and one parallel job for private projects. The practical difference between Azure DevOps and GitHub Actions for private work is narrow. Azure DevOps integrates more tightly with Azure Boards, Azure Repos, and Azure Artifacts if you are running an entirely Microsoft stack, but for a standalone pipeline the free tier limits are nearly identical.

Azure DevOps Server (the on-premises version, formerly Team Foundation Server) is free for up to 5 users under the Basic license and runs on any Windows Server machine you already own or manage. Pipelines run on your own hardware, so there are no minute limits. This is the right choice for teams with an on-premises server, air-gapped environments, or anyone who already runs a Windows Server and wants unlimited CI/CD minutes without a cloud account.

GitLab CI/CD at 400 minutes per month is tighter than GitHub Actions for private work, but GitLab’s self-hosted Community Edition (free, open source) runs the full GitLab platform — issue tracking, merge requests, container registry, and CI/CD pipelines — on your own server with no minute restrictions. Pairing GitLab CE with an OCI A1 VM gives you a complete DevSecOps platform for $0.

Self-hosted runners on GitHub Actions and Azure DevOps are the highest-leverage option for projects that have already provisioned a free cloud VM (OCI A1, GCP e2-micro). The setup is entirely cloud-based: SSH into the VM, download the runner agent, register it with your GitHub organization or repository, and start it as a systemd service. The agent runs permanently in the cloud, polling GitHub over outbound HTTPS for queued jobs — no local machine involved, no inbound firewall rules needed. A single OCI A1 instance (2 OCPU, 12 GB RAM) can comfortably handle concurrent pipelines across several repositories. For public repositories, self-hosted runners are completely free with no platform fee. For private repositories, GitHub began charging a $0.002/min platform fee in March 2026 regardless of where the runner is hosted — on OCI, AWS, GCP, or anywhere else. At $0.002/min, a 10-minute build costs $0.02; a team running 500 builds per month of 10 minutes each pays $10/month in platform fees, even with a free VM underneath.

Limitations

GitHub Actions’ 2,000 free minutes for private repositories apply to Linux runners only at 1× rate. Windows runners consume minutes at 2× and macOS at 10×. A 10-minute macOS build uses 100 of your 2,000 monthly minutes. Projects that need macOS builds (iOS, cross-platform Electron apps) should route those jobs to a self-hosted macOS machine and reserve hosted runner minutes for Linux-only steps.

Azure DevOps Services no longer grants free parallel jobs to new private projects by default — you must request the free grant from Microsoft, which requires manual approval and can take several business days. Check the Azure DevOps parallel jobs page before assuming the 1,800 minutes are immediately available on a new account.

Azure DevOps Server requires a Windows Server license, which is not free. If you already run Windows Server for other reasons, the DevOps Server layer is free. If you are standing up infrastructure specifically for CI/CD, GitLab CE or Jenkins on Linux (including OCI’s free A1 ARM instances) is a lower-cost starting point.

GitHub Actions self-hosted runners on private repositories are no longer fully free. Starting March 2026, GitHub charges a $0.002/min platform fee for every job minute run on a self-hosted runner in a private repo, regardless of where the runner is hosted. This fee does not apply to public repositories. For private projects where cost is the priority, Azure DevOps self-hosted agents and GitLab CE self-hosted runners remain genuinely unlimited and free.

Self-hosted runners introduce a maintenance surface that hosted runners do not have: OS updates, runner version upgrades, disk space management, and secrets management on the host VM. This is manageable on a cloud VM — set up the runner as a systemd service with automatic restarts and a weekly unattended upgrade cron job — but it is real operational overhead that a hosted runner eliminates entirely.

6. Database — Relational (SQL)

The Problem

Relational databases are where the free tier story gets complicated. A stateless frontend and a serverless backend can scale to zero and cost nothing when idle. A relational database cannot — it runs a persistent server process that must be available to respond to connections, whether or not any traffic exists. That persistent resource costs money. Most free tier options are time-limited trials, but Azure SQL is a notable exception: its free offer is permanent, with monthly usage limits rather than a calendar expiry.

Azure SQL Database offers the most generous always-free SQL tier of the three major cloud providers. The free offer runs on the General Purpose Serverless tier — up to 4 vCores, burstable — and resets at the start of each calendar month. Each Azure subscription can hold up to 10 free databases under this offer. The 32 GB data storage and 32 GB backup storage cover a real production workload for most small applications. When the 100,000 vCore-seconds monthly compute budget is exhausted, the database auto-pauses until the first day of the next month (this is the default behavior — you can opt into billing for overage instead if availability matters more than cost).

For projects that need a managed relational database outside Azure, Supabase and Neon are the practical always-free options. Both run PostgreSQL — the same engine as AWS RDS for PostgreSQL — so there is no migration friction if you later move to a paid managed option.

Supabase bundles Postgres with a REST API, real-time subscriptions, authentication, and storage in a single free project. For small applications, it removes the need for a separate backend service entirely. The 500 MB database limit is tight for data-heavy applications but sufficient for most early-stage projects.

Neon runs serverless Postgres that scales to zero, eliminating idle compute cost. 190 free compute hours per month covers intermittent development and low-traffic production use.

If you need Azure SQL beyond the free tier limits, Azure SQL Basic DTU at approximately $4.90/month is the cheapest always-on paid entry point — 5 DTUs and 2 GB of storage, no auto-pause risk. AWS RDS db.t3.micro runs $13–15/month after the 12-month trial. GCP Cloud SQL has no meaningful free tier and starts at roughly $7–10/month for the smallest instance.

Limitations

The Supabase free plan pauses inactive projects after one week of no activity. When a paused project receives its first request, it resumes — but this takes 10–30 seconds. The project can be kept active by scheduling a lightweight ping (a GitHub Actions cron job hitting any endpoint), but this is an extra operational step that a paid plan eliminates.

Neon’s 190 free compute hours per month are consumed faster than expected if the database handles queries continuously rather than intermittently. A project running 24/7 with no scale-to-zero exhausts 190 compute hours in roughly eight days.

The Azure SQL free tier auto-pauses the database when the monthly 100,000 vCore-second budget is exhausted — all connections are refused until the first of the following month. For a development or low-traffic production database this is often acceptable, but any application that must stay available throughout the month (rather than just during business hours or intermittent use) should either enable the “bill for overage” option or upgrade to Azure SQL Basic DTU (~$4.90/month), which runs without a compute cap. Basic DTU (5 DTUs) is designed for light OLTP workloads with predictable, low concurrency; upgrade to Standard S0 (10 DTUs, ~$15/month) if you see consistent query timeouts under normal load.

7. Database — NoSQL

The Problem

Not every application needs a relational schema and ACID transactions. For applications that store user-generated content, session data, product catalogs, or event logs — where the shape of records varies and joins are rare — a NoSQL document database often fits better and costs less. The always-free NoSQL tiers are the most generous in the entire database category.

AWS DynamoDB is the most scalable always-free NoSQL option: 25 GB of storage and enough provisioned throughput to handle 200 million requests per month. The free tier applies to provisioned capacity mode only — on-demand tables are charged per request from the first operation. DynamoDB’s free tier does not expire; it is not a 12-month offer.

Azure Cosmos DB offers 1,000 RU/s (request units per second) and 25 GB of storage, permanently. 1,000 RU/s translates to roughly 1,000 point reads or 100 simple queries per second — sufficient for a backend API serving a few hundred concurrent users without bursts. Cosmos DB supports multiple APIs (document, key-value, graph, column-family), making it adaptable to different data models without changing providers.

GCP Firestore has the tightest free tier: 50,000 reads and 20,000 writes per day with 1 GB of storage. For apps that batch-read data or cache aggressively, this is workable. For real-time apps that stream frequent small reads — chat, live dashboards — the read limit can be exhausted in minutes by active users.

MongoDB Atlas M0 (shared cluster, 512 MB) is a cloud-agnostic option deployable on AWS, Azure, or GCP. It is free indefinitely for a single cluster, and MongoDB’s query model is the most familiar to developers coming from a relational background.

Limitations

DynamoDB’s access pattern constraints are the most common source of friction when migrating from a relational database. DynamoDB does not support ad-hoc queries across all records without a full table scan — queries must target a partition key. Designing the data model to match access patterns upfront is required. This is not a limitation of the free tier specifically; it is fundamental to how DynamoDB works, and misunderstanding it leads to either expensive scans or a difficult data model redesign later.

Cosmos DB’s 1,000 RU/s free limit is shared across all containers (collections) in the free account. If your application creates multiple containers, the 1,000 RU/s is divided among them — not multiplied. A single container doing intensive reads can starve others.

Firestore’s daily limits (rather than monthly) mean a single traffic spike can consume the entire day’s quota and return errors for the remainder of the day. Unlike bandwidth limits on hosting services, Firestore returns errors — not degraded responses — when the daily limit is hit. Applications relying on Firestore should implement client-side retry logic and read caching to handle limit exhaustion gracefully.

8. ML & Data Analytics

The Problem

With AI-assisted development now mainstream, many new applications include an ML component — calling an LLM, generating embeddings, classifying images, or processing data at scale. Each major cloud provider has built a significant ML platform, and the free tier picture here is more uneven than any other layer: the most useful offerings (pre-built model inference APIs) have the thinnest free tiers, while data analytics gets surprisingly generous always-free limits.

GCP BigQuery stands out as the only always-free large-scale analytics service among the three providers. 1 TB of free query processing per month covers substantial workloads — querying a 1 GB table 1,000 times, or a 100 GB table 10 times. For analytics-heavy workloads, GCP’s free tier is materially more generous than AWS or Azure.

AWS Athena and Azure Synapse both charge from the first query. Athena at $5 per TB scanned adds up quickly for exploratory analysis on large datasets — a common pattern in data science and analytics workflows.

Limitations

BigQuery’s 1 TB free query limit applies per billing account per month and does not carry over. A single poorly-written query scanning a 2 TB table exhausts double the monthly free allocation in one run. BigQuery offers a “dry run” mode that estimates bytes scanned before executing — use it before running anything on a large table.

Gemini’s free tier rate limits (15 requests per minute for 2.0 Flash) are sufficient for development use but will throttle any production application with concurrent users. Burst traffic exceeding the rate limit returns HTTP 429 errors. Applications using the Gemini free tier need client-side retry logic with exponential backoff. The free tier is not appropriate for applications with response-time SLAs or high simultaneous user counts.

A complete stack — React frontend on GitHub Pages, serverless API on AWS Lambda + API Gateway, and data in DynamoDB — runs entirely within always-free limits. If you need a relational SQL database, Azure SQL’s free offer (100,000 vCore-seconds/month, 32 GB) keeps it within the always-free envelope on Azure; Supabase or Neon work for any stack and also cost nothing.

The one cost you will almost certainly not avoid is a domain name. Every provider gives you a free subdomain (*.azurestaticapps.net, *.cloudfunctions.net, *.pages.github.io), but a custom domain runs $10–15/year. That is often the entire annual infrastructure cost for a small production app.

The biggest risk is not the free tier limits themselves — it is not knowing when you have left them. Every provider makes it easy to accidentally provision a paid service: a VM left running, an RDS instance with no traffic, a container registry that forgot to prune old images. Set a billing alert at $1 per month in whichever provider’s console you use. If it fires, something slipped outside the free tier and you want to know about it before the bill is $50.
